# `/game/volley` 구현 전 이해 보고서

## 1. 요청 이해 요약
- 목표 게임: `/game/volley`에서 실행되는 1:1 실시간 미니게임.
- 레퍼런스: `피카츄 배구`와 유사한 규칙.
- 핵심 조작: 기존 플랫포머(로비/던전)의 이동 및 점프 감각을 최대한 그대로 사용.
- 경기 시작 조건: 양 플레이어가 각자 `10코인`을 먼저 베팅(입장비)해야 시작.
- 승리 조건: 공을 상대 진영 바닥에 닿게 하면 득점.
- 패배 조건: 내 진영 바닥에 공이 닿으면 실점.
- 최종 승리 조건: `5점 선취` 시 즉시 승리.
- 코인 보상 규칙: 승자가 베팅 코인을 획득(총 20코인 풀 기준, 패자는 베팅분 소진).
- 필드 조건:
  - 중앙에 라켓(네트 역할)이 고정됨.
  - 플레이어는 중앙 라켓 기준 좌/우 진영에서 시작.
  - 외곽 아웃라인 없음.
  - 좌/우 벽이 있어 공이 벽에 맞아 반사됨.
- 충돌 규칙: 플레이어 몸과 공 충돌로 공을 튕길 수 있어야 함.
- 물리 감각: 공은 탄성이 높고 반발이 명확해야 함.

## 2. `division.md` 기준 현재 상태 확인
- `/game` 페이지는 현재 허브 뼈대만 있고 실제 미니게임 루프는 미구현.
- 미니게임 진입은 이미 구현됨:
  - `minigame_invite` -> `minigame_invite_response` -> `minigame_start`
  - 클라이언트 이동: `/game?session=<session_id>`
- 분업 규칙상 필수:
  - 승패/정산(코인 포함)은 클라이언트 확정 금지.
  - 최종 정산은 서버에서 처리하고 DB(MongoDB) 반영 필요.

## 3. 기존 플랫포머 이동 재사용 대상(확인 완료)
- `static/lobby.js`, `static/dungeon.js`에 동일 계열 이동 물리 존재:
  - 중력/가속/최대속도/지상·공중 마찰
  - 가변 점프(짧게/길게), 점프 컷
- 따라서 `/game/volley` 플레이어 조작은 위 상수/로직을 재사용해 체감 일관성을 맞출 수 있음.

## 4. 게임 규칙을 구현 단위로 재정의
1. 매치 시작 전 서버가 양 플레이어 코인 보유량을 확인하고, 각자 10코인을 차감(에스크로)한다.
2. 라운드 시작 시 공은 중앙 상단 또는 직전 득점 반대편에서 서브.
3. 공이 바닥과 충돌하면 충돌 지점의 진영에 따라 득점자 결정.
4. 득점 후 짧은 리셋(플레이어 위치/공 위치 초기화) 후 다음 랠리 시작.
5. 한쪽이 5점에 먼저 도달하면 매치 종료(즉시 승리 확정).
6. 종료 시 서버가 승자에게 베팅 풀(20코인)을 지급하고 결과를 DB에 반영한다.

## 5. 점수 UI 명세
- 점수는 숫자로 크게 표시한다.
- 좌측 플레이어 점수는 좌측 진영 상단 중앙, 우측 플레이어 점수는 우측 진영 상단 중앙에 배치한다.
- 스타일 기준:
  - 큰 폰트 크기(게임 내 최우선 가시성)
  - 배경과 명확히 대비되는 색상
  - 랠리 중에도 항상 고정 표시(숨김 없음)
- 표기 예시: `좌측 3 : 2 우측`이 아니라, 각 진영 위에 각각 `3`, `2`를 독립적으로 크게 노출.

## 6. 물리/충돌 요구사항 명세(구현 전 합의안)
- 공:
  - 원형 콜라이더 사용.
  - 바닥/벽/천장/라켓/플레이어에 대해 반사 처리.
  - 반사 계수(restitution)는 높게 설정(예: 0.85~0.95 범위에서 튜닝).
  - 최소/최대 속도 클램프를 둬서 멈춤/폭주 방지.
- 플레이어-공 충돌:
  - AABB(플레이어) vs Circle(공) 충돌 보정.
  - 플레이어 속도(`vx`, `vy`)를 공 반사에 일부 가산해 타격감 강화.
- 필드:
  - 중앙 라켓은 고정 장애물.
  - 좌/우 벽은 반사 벽.
  - 바닥은 득점 판정면.

## 7. 서버/클라이언트 책임 분리(division 규칙 반영)
- 클라이언트:
  - 입력 처리, 렌더링, 임시 예측.
  - 점수 대형 UI 렌더링.
- 서버:
  - 입장 전 코인 검증(각 10코인 이상).
  - 매치 시작 시 10코인씩 차감.
  - 점수, 승패, 코인 정산 최종 확정.
  - 세션 단위 상태 관리(진행/종료).
- 권장 이벤트(추가 필요):
  - `volley_entry_lock`(10코인 차감 잠금), `volley_round_start`
  - `volley_score_update`, `volley_match_end`
  - `volley_settlement`(승자 지급 포함 정산 결과)

## 8. 확인된 리스크와 대응
- 리스크: 양쪽 클라이언트 물리 오차로 공 위치 불일치.
  - 대응: 서버 스냅샷/보정 주기 도입 또는 서버 권위 물리로 전환.
- 리스크: 고탄성 설정 시 터널링(충돌 누락).
  - 대응: 고정 timestep + 연속 충돌 보정(서브스텝) 적용.
- 리스크: 코인 정산 클라이언트 조작.
  - 대응: 베팅 차감/승자 지급 모두 서버에서만 수행.
 - 리스크: 경기 도중 이탈 시 베팅 정산 애매함.
  - 대응: 이탈 패배 처리 규칙 또는 환불 규칙을 서버 정책으로 고정.

## 9. 결론
- 요청하신 게임 의도는 다음으로 이해함:
  - "플랫포머 조작 감각을 유지한 1:1 피카츄 배구형 바운스 게임"
  - "중앙 라켓 + 벽 반사 + 몸 충돌 + 바닥 득점"
  - "각자 10코인 베팅 후 시작, 5점 선취 승리"
  - "점수는 각 진영 상단에 큰 숫자로 상시 표기"
  - "승패/보상은 서버 권위"
- 위 이해를 기준으로 바로 구현을 시작하면 됨.
